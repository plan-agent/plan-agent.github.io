这份文档的作用是：把需求转成“开发结构”。

```md
# docs/plan.md

## 1. 项目目标
在模块三中完成一个教学版 Web MVP，让学生练习：
- 从 spec 到 plan
- 从 plan 到 tasks
- 使用 Claude Code / Codex 分步实现
- 用最小测试保证结果可验证

## 2. 技术选型

### 前端
- Next.js App Router
- TypeScript
- Tailwind CSS

### 数据
- `data/registrations.json` 作为教学版持久化文件
- `data/faq.json` 作为 FAQ 本地知识库

### 测试
- Vitest
- 手动验证清单

## 3. 选型理由
- 单仓库即可完成前后端演示
- 不依赖外部账号和云服务
- 页面与 API 可以一起讲
- 业务逻辑可以从页面中抽离，便于测试
- 适合作为“从点子到 MVP”的课堂案例

## 4. 页面与路由设计
- `/`：首页，展示活动信息和两个入口按钮
- `/register`：报名页，表单提交
- `/faq`：FAQ 页，常见问题与答疑
- `/admin`：管理员页，查看报名记录
- `/demo`：路演页，展示项目价值和统计数据
- `/api/registrations`：处理报名提交

## 5. 业务模块设计

### 5.1 首页模块
- Hero 区
- 活动亮点区
- 四大模块介绍区
- CTA 区

### 5.2 报名模块
- 前端表单组件 `RegistrationForm`
- 校验逻辑 `lib/validation.ts`
- 数据写入逻辑 `lib/registrations.ts`
- API 路由 `/api/registrations`

### 5.3 FAQ 模块
- 本地 FAQ 数据 `data/faq.json`
- 匹配逻辑 `lib/faq.ts`
- 页面组件 `FaqAssistant`

### 5.4 管理员模块
- 从 `lib/registrations.ts` 读取所有报名记录
- 默认按时间倒序展示

### 5.5 路演模块
- 从报名数据计算：
  - 报名总人数
  - 角色分布
  - 有创意想法的人数
- 通过 `lib/stats.ts` 输出统计结果

## 6. 核心数据流
1. 用户访问首页，了解活动信息。
2. 用户进入报名页并填写表单。
3. 前端先做基础校验。
4. 校验通过后调用 `/api/registrations`。
5. API 再次校验，并将数据写入 `data/registrations.json`。
6. 管理员页读取报名数据并展示。
7. 路演页复用同一份数据生成统计结果。

## 7. 代码组织原则
- 页面只负责展示和交互
- 核心业务逻辑放在 `lib/`
- 数据文件放在 `data/`
- 测试优先覆盖 `lib/` 中的函数
- 尽量少引入新依赖

## 8. 里程碑拆分

### M1：项目骨架
- 初始化项目
- 创建基础路由
- 写入 docs 文档

### M2：首页与静态页面
- 完成首页 UI
- 完成 FAQ 页和 Demo 页的静态版本

### M3：报名表单
- 完成表单 UI
- 完成前端校验
- 完成 API 提交
- 完成数据写入

### M4：FAQ 答疑逻辑
- 完成 FAQ 数据文件
- 完成关键词匹配逻辑
- 页面可返回答案

### M5：管理员页与统计
- 管理员页读取报名记录
- 路演页显示统计数据

### M6：测试与收尾
- 完成核心函数测试
- 整理手动验证清单
- 准备 demo script

## 9. 风险与应对
- 风险：课堂时间有限  
  应对：先确保首页、报名、FAQ 跑通，管理员页和统计页可后补
- 风险：学生环境不同  
  应对：统一使用教学版本地数据文件
- 风险：AI 一次改动过大  
  应对：强制按里程碑逐步实现，每步都验证

## 10. 模块四可升级方向
- 把 FAQ 从关键词匹配升级为真实 LLM 问答
- 把本地 JSON 升级为数据库
- 增加管理员登录与导出功能
- 增加消息通知与自动分组
