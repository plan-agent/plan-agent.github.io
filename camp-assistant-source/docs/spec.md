# docs/spec.md

## 1. 项目名称
双创营报名与答疑助手 Web MVP

## 2. 一句话定义
一个帮助双创营主办方统一展示活动信息、收集报名、回答常见问题，并在路演时展示结果数据的轻量 Web 应用。

## 3. 背景问题
当前活动信息可能散落在海报、微信群、聊天记录和问卷链接中，主办方存在以下痛点：
1. 活动介绍不集中，用户获取信息成本高。
2. 报名数据分散，后续统计和筛选不方便。
3. 高频问题需要人工重复回答。
4. 路演时缺少“产品化结果页”展示项目价值。

## 4. 目标用户
- 用户 A：准备报名的学生
- 用户 B：活动主办方 / 管理员
- 用户 C：路演评委 / 导师 / 合作方

## 5. MVP 目标
在一个代码仓库内完成一个可运行、可演示、可路演的 Web MVP，至少覆盖：
- 活动信息展示
- 报名表单提交
- FAQ 答疑
- 管理员查看报名记录
- 路演结果页

## 6. 用户故事
- US-01：作为准备报名的学生，我希望快速了解活动介绍、时间、对象和课程模块，以便决定是否报名。
- US-02：作为准备报名的学生，我希望在线提交报名信息，并获得成功反馈。
- US-03：作为准备报名的学生，我希望能快速查到“是否需要编程基础”“是否可跨专业组队”等常见问题。
- US-04：作为主办方，我希望查看已经提交的报名记录，便于统计和后续联系。
- US-05：作为主讲人/路演者，我希望有一个结果页，能展示项目解决的问题、核心功能和简单运营数据。

## 7. 功能需求

### FR-01 活动信息展示页
首页需要展示：
- 活动名称
- 一句话介绍
- 活动亮点
- 时间 / 地点 / 招募对象
- 四大模块简介
- CTA 按钮（立即报名 / 常见问题）

验收标准：
- 首页在移动端和桌面端均可正常浏览
- 点击“立即报名”进入 `/register`
- 点击“常见问题”进入 `/faq`

### FR-02 报名表单
报名表单至少包含以下字段：
- 姓名（必填）
- 学校（必填）
- 专业（必填）
- 年级（必填）
- 邮箱（必填）
- 角色倾向（必填：技术开发 / 产品运营 / 设计传播 / 暂未确定）
- 技能标签（选填）
- 创意想法（选填，不超过 200 字）
- 参加期待（选填，不超过 200 字）

交互要求：
- 必填字段缺失时给出中文错误提示
- 邮箱格式错误时给出中文错误提示
- 提交成功后显示成功消息
- 提交后的记录可在管理员页看到

验收标准：
- 有效数据可成功提交
- 无效数据会被拦截并提示
- 新提交记录写入数据文件

### FR-03 FAQ 答疑
FAQ 页需要提供：
- 一个输入框
- 若干常见问题快捷按钮
- 一个答案展示区域

回答机制：
- 先使用本地 FAQ 数据
- 用关键词匹配返回最相关答案
- 若未匹配，返回统一兜底提示：
  “当前未找到精确答案，请联系主办方或查看活动介绍页。”

建议内置问题：
- 报名截止时间是什么时候？
- 没有编程基础可以参加吗？
- 可以跨专业组队吗？
- 需要自带电脑吗？
- 双创营结束后会有什么成果？

验收标准：
- 输入常见问题时能返回预置答案
- 输入未知问题时能返回兜底提示

### FR-04 管理员查看报名信息
管理员页 `/admin` 需要展示：
- 报名总人数
- 最近报名记录列表
- 每条记录的核心字段：姓名 / 学校 / 专业 / 角色倾向 / 提交时间

教学版要求：
- 不做登录鉴权
- 不做复杂筛选
- 以“可演示”为主

验收标准：
- 页面能读取已有报名数据
- 提交新报名后刷新管理员页可以看到新记录

### FR-05 路演页 / 结果页
路演页 `/demo` 需要展示：
- 项目要解决的问题
- 产品方案
- 核心功能概览
- 简单数据看板：
  - 报名总人数
  - 不同角色人数分布
  - 填写了创意想法的人数
- 下一步升级方向

验收标准：
- 路演页可直接用于演示
- 数据看板来自当前报名数据

## 8. 数据模型

```ts
type Registration = {
  id: string;
  name: string;
  school: string;
  major: string;
  grade: string;
  email: string;
  role: "技术开发" | "产品运营" | "设计传播" | "暂未确定";
  skills?: string[];
  idea?: string;
  expectation?: string;
  createdAt: string;
}
## 9. 非功能要求
- 页面文案为中文
- 界面简洁，适合课堂演示
- 代码结构清晰，便于 AI agent 阅读与修改
- 业务逻辑尽量放入 lib/，方便测试
- 教学版优先本地运行，不追求生产部署

## 10. 范围外
- 用户登录/注册
- 支付系统
- 邮件/短信通知
- 真实 LLM 接口
- 云数据库与权限管理
